Migrate active owned apps from Sentry to Canary.

Every active deployed app in the phrazzld and misty-step portfolios must have Canary error ingest, uptime/health coverage, deployed readback proof, and no remaining runtime Sentry dependency before the Sentry account can be cancelled.

Plan date: 2026-06-16

Primary repo: canary (this repository)

Goal status: active

Cancellation rule: cancel Sentry only after zero active-app dependency proof and an account export/receipt.

Acceptance

Each active deployed app has a passing integration receipt: health route or monitor, Canary target/check-in, synthetic error ingest readback, deployed smoke, and Sentry code/env removal proof.

Verify

Use bin/canary integrate status, bin/canary dogfood audit --strict --json, Vercel/Fly deploy checks, production health probes, and bin/canary errors <service> readback.

Communicate

Report per-app evidence, dirty-state conflicts, secrets/account blockers, review results, deploy status, and final Sentry cancellation state.

Stop If

Stop before irreversible cancellation if any active app still references Sentry, any live app lacks Canary proof, or Sentry billing cancellation requires an out-of-band human checkpoint.

Active App Boundary

An app is active when it has a non-archived GitHub repo or a non-deprecated Vercel/Fly production deployment with a public URL. Archived repos with still-live Vercel projects remain active until the deployment is removed or explicitly retired.

Execution Sequence

  1. Produce a machine-readable inventory from Vercel, Fly, GitHub, local checkouts, and Canary dogfood registry.
  2. For each app, run integrate discover/status/plan and record gaps.
  3. Patch one app at a time on a feature branch, respecting dirty worktrees and repo-specific AGENTS.md.
  4. Deploy and verify health plus synthetic error readback in Canary.
  5. Remove Sentry packages, config files, source map upload settings, docs that make Sentry primary, and platform env vars.
  6. Run portfolio audit: no active-app Sentry references, no active-app missing Canary proof.
  7. Export Sentry account/project state if possible, then cancel Sentry through CLI/browser only after proof is green.

Review Focus

  • Security: never expose Canary ingest keys in client bundles except deliberately scoped public/browser keys.
  • Reliability: health checks must reflect app readiness, not just static 200 responses.
  • Completeness: Sentry removal must cover package deps, Next config wrappers, instrumentation hooks, docs, workflows, and Vercel env names.
  • Account safety: Sentry cancellation is irreversible enough to require final zero-dependency evidence.
