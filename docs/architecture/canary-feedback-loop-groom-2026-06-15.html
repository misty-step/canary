Canary feedback loop groom

Ambitious product and architecture direction for dogfooding Canary, watching Canary itself, proving value to consuming applications, and connecting Canary to auto-triage without making Canary an agent runtime.

Thesis

DecisionCanary should remain the deep observability and context module: ingest, health, incidents, timelines, claims, annotations, query/read models, webhooks, and agent-native inspection. Auto-triage should run as a Bitterblossom workload over Canary's stable interfaces, not inside Canary and not as a separate bespoke microservice until Bitterblossom proves insufficient.

GapThe current product feedback loop can say "covered" and "unhealthy", but it does not yet produce a durable value receipt that answers: what did Canary detect, who or what acted, what changed, and how did Canary prove the fix worked?

Live Evidence

Self-watch is working but noisy

bin/canary summary --json reported status: unhealthy with canary-watchman down. bin/canary doctor --json showed /healthz ok, /readyz ready, one pressured worker, stale witness check-in, and an open Canary incident.

Dogfood registry drift is visible

bin/canary dogfood audit --strict --json failed with 35 strict failures: unregistered deployments, stale registry evidence, and completed-ticket next actions. The tool catches drift, but the next-action loop is not yet closed.

Agent surfaces exist

The CLI exposes summary, services, errors, incidents, timeline, targets, monitors, dogfood, events, claims, integrate, doctor, and MCP manifest. OpenAPI includes agent guidance for webhooks, timeline replay, incident detail, claims, annotations, and telemetry.

Integration still overclaims

Reviewer evidence found receipts can be marked verified after target/key creation without synthetic ingest or check-in readback, global webhook presence can satisfy coverage, and browser capture still lacks a safe public relay/DSN story.

External Prior Art

Google SRE Workbook, alerting on SLOs Alert on user-impacting budget burn, not every threshold crossing. Multi-window burn-rate alerts balance precision, recall, detection time, and reset time. Add SLO/error-budget views and alert-plane oracles. Health checks alone are too shallow.
Sentry Seer docs The observability product provides rich issue context and ownership controls; AI debugging can hand off to external coding agents. Canary should own context and claim protocol; Bitterblossom should own reactive code-changing work.
OpenTelemetry observability docs Shared signal language matters; logs, metrics, traces, and events should have clear semantic boundaries. Keep Canary's native/bridge split. Do bounded operational events and links, not SQLite as a telemetry warehouse.

World-Class Target

  1. Agent control tower: one command returns overall verdict, blocking signals, witness age, worker pressure, dogfood gap count, open canary incidents, receipt links, and next operator action.
  2. Dogfood value receipt: every owned service has a current receipt showing coverage, current health, errors/incidents, active claim owner, last triage action, and last verified outcome.
  3. SLO and alert-plane reliability: Canary distinguishes "route is ready" from "Canary can still alert and create useful work"; witness fails on alert-delivery impairment and sustained pressure.
  4. Safe responder boundary: responder agents get service-bound read plus narrow responder-write authority, redacted context manifests, receiver conformance tests, and read-audit events.
  5. Closed integration loop: integration cannot claim verified until platform env, health, target/monitor, synthetic ingest/check-in, service-specific webhook, and query readback have all been proven.
  6. Bitterblossom triage workload: signed webhook wakes Bitterblossom, Bitterblossom replays timeline, claims incident/error group, investigates in repo, writes annotations/evidence, and verifies via Canary readback.
