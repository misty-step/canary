Self-Watch Operator Verdict

Make bin/canary doctor answer the first operator question: can Canary watch itself right now, what evidence proves that, and what should the next agent do without opening a dashboard?

Chosen Design

Add one derived doctor.verdict object assembled from the existing doctor probes. Keep HTTP routes, store writes, workers, and the witness runtime unchanged.

This keeps the interface deep: callers ask one question and receive a compact control-tower answer, while detailed probe evidence remains available underneath.

Outcome

  • doctor --json includes verdict.overall, blocking signals, witness age, open Canary incident, worker pressure, dogfood gap count, receipt/run refs, and next action.
  • doctor text prints a compact verdict and keeps existing probe lines.
  • Worker pressured means ready with operational pressure, not double-counted as failed.

Assumptions

  • The CLI may shell out for operator evidence, as it already does for DR and dogfood inventory.
  • GitHub Actions witness run discovery is best-effort and must fail open with the replacement command.
  • Open incidents for service canary are self-watch blockers; unrelated service incidents stay visible below the verdict.

Stop Conditions

  • The verdict can be produced without a new server route or schema migration.
  • Fixture tests cover stale witness, worker pressure, MCP drill-down names, and live-shape actionability.
  • Live bin/canary doctor --json and text output produce usable operator guidance.

Risks

  • Clock parsing: witness age needs RFC3339 parsing and must degrade to null rather than panic.
  • GH auth: witness run discovery may be unavailable locally; verdict must still include commands.
  • False health: a ready route with down witness is not healthy; the verdict must prefer self-watch evidence.
