Backlog 052 + 057 / Factory Canary Lane

Ship a real Canary MCP server without creating a second semantic API.

Canary already owns the agent contract in the Rust CLI: tool_manifest(), JSON envelopes, scoped HTTP operations, and deterministic summaries. The chosen design is a stdio MCP adapter inside canary that translates the generated CLI manifest into MCP tools/list shape and dispatches tools/call through the same CLI-backed functions.

Quality System

Outcome: an MCP client can initialize, list Canary tools, and call a read-only tool.

Standards: one generated tool source, no LLM on request path, no new responder authority, no stale manifest.

Proof: MCP JSON-RPC smoke, manifest parity test, focused Rust tests, repo gate.

Critic: fresh diff-only review focused on protocol drift, auth leakage, and shallow adapters.

Stop: halt if the wrapper needs a parallel tool registry or claims responder safety beyond current scopes.

Current State

bin/canary mcp-manifest emits a CLI-backed manifest, but the checked-in snapshot is stale and no MCP server runs.

Change Shape

Add canary mcp-server, a smokeable stdio JSON-RPC server, generated tools/list, dispatch for existing manifest tools, docs, and snapshot parity.

Out Of Scope

No new responder-write scope, no browser capture authority, no autonomous repo mutation, no separate MCP-only business logic.

Verification System

  • Claim: Canary exposes a runnable MCP surface over the CLI contract.
  • Falsifier: tools/list differs from the generated manifest or tools/call cannot invoke a read-only command.
  • Driver: feed initialize, initialized, tools/list, and tools/call JSON-RPC lines into canary mcp-server.
  • Grader: MCP result has inputSchema, text content, structured content, and no secret leakage.
  • Evidence: checked test fixture and command transcript in closeout.

Landmark Integration

Add a release workflow that waits for Canary CI success on master, checks out full history, and runs misty-step/landmark@v1 with GH_RELEASE_TOKEN, optional OPENROUTER_API_KEY, Node 24, and synthesis enabled but non-blocking.

Bitterblossom Trigger Contract

Document the event path: signed Canary webhook is wake-up only; responder verifies signature and delivery id, queries Canary for timeline/report/context, creates a remediation claim, acts outside Canary, then writes annotations and claim evidence back.

Risk Focus

MCP is allowed to be an adapter, not an excuse to widen authority.

Tool errors should be MCP tool results with isError; protocol errors should stay JSON-RPC errors. Secret-bearing config values must never appear in responses or docs.
