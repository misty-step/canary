BACKLOG 047 · CHILD #6 · ALERT-PLANE SLI TRAJECTORY

An agent reading one bounded report can see whether each service's availability and error trajectory is worsening — without Canary pre-deciding urgency.

Children 1–5 shipped the alert-plane verdict, check-in skew policy, induced-impairment rehearsal, windowed SLI read models, and default SLO classes. This final slice makes those SLIs visible and directional: surface the already-computed windowed SLIs + SLO targets on the CLI (which silently drops them today), and add a deterministic, low-N-safe trajectory delta — this window vs the prior equal-length window. It deliberately does not build a burn-rate-vs-budget ratio or a page/ticket severity verdict: that math assumes high-traffic request streams Canary does not have, and that verdict is policy the responder owns.

CHOSEN DESIGN

Windowed SLIs + SLO targets + signed trajectory delta on /api/v1/report & CLI. Two budget signals kept separate (availability ratio, raw error count). Regenerate the MCP manifest snapshot; defer the runnable server to 052.

QUALITY SYSTEM

Standard: a delta is evidence (a fact about two windows); never a budget promise or an urgency policy. Sub-floor windows say insufficient_samples, never a fake swing.

Proof: focused Rust tests (store/server/CLI/manifest-parity) + production-image report readback + a negative grep oracle that no severity field leaked + one fresh cross-family contract critic.

SCOPE

Goal: let an agent judge "is this service getting worse?" from one bounded read, without the substrate deciding how urgent that is.

Non-goals: MWMBR burn-rate ratio against an absolute budget; any page/ticket/recommended_severity field; webhook payload severity or delivery routing (stays severity-agnostic); new query windows (5m/30m/3d); a per-service SLO config table; a runnable MCP server (ticket 052); Prometheus burn-rate gauges on /metrics.

Invariants: RFC 9457 problem details on invalid window; deterministic summaries, no LLM on request path; single SQLite writer — the prior-window lookup is a bounded extra read, never a second writer or a long-held lock; one semantic contract across report/CLI/manifest (enforced by the parity test).

DESIGN — the trajectory primitive

Compute the current window summary as today, then compute the immediately-preceding equal-length window by re-running the read model at now − duration(window). Difference the two. Keep the two failure modes as separate raw signals — do not synthesize one number. Illustrative shape (the builder owns exact field names/types):

trajectory: {
  availability_delta: Option<f64>,     // signed; current_ratio − prior_ratio (negative = worse)
  prior_availability_ratio: Option<f64>,
  error_total_delta: Option<i64>,       // signed; current_total − prior_total
  prior_error_total: u64,
  sample_basis: u64,                    // min(current, prior) checks/check-ins backing the delta
  status: "ok" | "insufficient_samples" // sub-floor → deltas null, no fake swing
}

Why a delta, not a burn rate: a burn-rate-vs-budget ratio is policy ("budget consumed too fast" against an absolute the substrate doesn't own) and, on N≈60–120 samples/hour, a single failed probe produces a huge ratio (the SRE Workbook's own low-traffic caveat). A signed delta vs the prior equal-length window survives low N, stays a fact, and is exactly the "is it worsening?" input an agent needs to decide its own next action. Why drop the events/hour budget: error_budget_events_per_hour conflates the probe stream with the error-ingest stream and is too jumpy at this cadence — surface the raw error-count delta instead and let the SLO target travel as context.

VERIFICATION SYSTEM

Claim: an agent sees worsening availability/error trajectory from one bounded report; the substrate emits no urgency policy.

Falsifier: a service whose 1h availability is far below its prior 1h but the report shows no delta; or a sub-floor window emitting a non-null delta that flaps; or any page/ticket/recommended_severity field on a surface.

Driver: focused Rust tests (store current-vs-prior with injected clock; serializer; CLI render; manifest parity) + production-image report readback.

Grader: report & CLI JSON envelopes contain trajectory with correct sign and insufficient_samples at the floor; grep proves no new severity field beyond the pre-existing incident severity.

Evidence packet: readback JSON + test output linked from the implementation PR.

Cadence: unit/parity tests every gate; production-image readback before merge and after any floor change.

Gaps/waiver: cadence-aware (per-target probe-rate) floor deferred; fixed floor accepted for this slice.
