dogfood value receipts execution plan

SOURCE: backlog.d/046-dogfood-value-receipts.md

Make dogfood coverage produce per-service value receipts.

Add the smallest CLI read model that answers what Canary currently proves for one registered service, starting with linejam and chrondle. The receipt should combine existing coverage, health, errors, incidents, claims, annotations, telemetry events, and verification status without inventing a new storage system.

CHOSEN DESIGN

Add bin/canary dogfood value --service <name> as a CLI read model over existing live probes and registry data.

QUALITY SYSTEM

Tier: substantive. Public CLI and operator workflow behavior change.

Proof: failing CLI test, focused Rust tests, live value receipts for pilots, doctor JSON summary, GitHub CI, fresh-context review.

Baseline: strict audit reports covered=4, partial=33, blocked=5, ignored=1, strict_failures=45.

Stop if the receipt needs new persistence, new auth scopes, or cross-service mutation.

STANDARDS

Agent-readable JSON, deterministic text summary, no secret output, no writes, and no responder-boundary creep.

ACCEPTANCE

Linejam and chrondle receipts distinguish current value proof, explicit absence, and one next action.

VERIFY

bin/canary dogfood value --service linejam --json, chrondle, targeted tests, strict gate.

COMMUNICATE

Report baseline gaps, files changed, live receipts, critic result, gate result, and clean tree.

STOP IF

Live API auth missing, registry cannot identify service, strict gate fails for unrelated breakage, or reviewer finds a blocking public-surface issue.

CURRENT STATE

bin/canary dogfood audit --strict --json fails with 45 strict failures. Existing dogfood audit proves coverage counts, but not value outcome per service.

CHANGE SHAPE

Extend canary-cli dogfood subcommands, add receipt builder/summarizer/tests, update docs, MCP manifest, and doctor verdict value summary.

PROOF SURFACE

CLI transcript JSON/text, fixture tests in crates/canary-cli, and final doctor JSON showing value coverage counts.

SCOPE

Goal: ship a first useful value receipt loop, not close every dogfood deployment gap.

Non-goals: dashboard UI, new database tables, auto-remediation, broad registry cleanup, or arbitrary-user responder authorization.

Invariants: read-only CLI behavior, scoped API keys, deterministic summaries, and responder boundary remain intact.
