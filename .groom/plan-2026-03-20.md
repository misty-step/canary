# Grooming Plan — 2026-03-20

## Theme: Dogfooding Wave

Get all owned services reporting to Canary. Learn what's missing from real usage.

## Key Decision

**API-first, not SDK-first.** The ingest API is 3 required fields (`service`, `error_class`, `message`).
Health target registration is 2 fields (`name`, `url`). Any HTTP client in any language works.
No new SDKs. Existing TS/Elixir SDKs are optional convenience wrappers, not blockers.

## Architecture Assessment

Canary handles 8+ services with zero structural concerns:
- SQLite pool_size:1 is fine for this ingest volume
- Health checker scales linearly (one GenServer per target, jitter prevents thundering herd)
- Dashboard handles 50+ rows comfortably
- Status endpoint returns one row per service — trivial at 8
- N+1 in Query.health_status/0 is ~17 queries at 8 targets, all sub-ms on SQLite
