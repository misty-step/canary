# Grooming Plan — 2026-03-22

## Theme: Agent-First Platform

Transform Canary from a data recorder into an intelligence layer where AI agents
are the primary consumer. One call to answer "what's wrong?", auto-correlated
incidents, semantic classification, full-text search.

## Vision Lock

**World-class, AI-first uptime monitoring and error logging.**
Everything oriented towards making it easy for agents to work with.

## Key Decisions

1. **No MCP server yet.** Fix the API surface first — MCP is a transport layer on top of good APIs.
2. **Incidents as the primary object.** Errors and health events are signals feeding into incidents, not standalone entities.
3. **Deterministic over LLM.** Classification is rule-based. Summaries are templates. No LLM on request path.
4. **One call, one response.** The report endpoint replaces 7 scattered endpoints as the agent's primary interface.

## Architecture Assessment

Four parallel critiques (Ousterhout, Carmack, Grug, reference search) converged:
- Internals are solid — StateMachine, Ingest, Grouping, webhook pipeline all well-designed
- API surface is the #1 problem — agents need 3+ calls across 7 endpoints
- No incident concept — errors and health are disconnected streams
- Market gap: no self-hosted, lightweight, agent-first monitoring exists

## Market Position

Canary occupies an empty niche. Competitors are either:
- SaaS giants with AI bolted on (Datadog, Sentry)
- Self-hosted with zero agent features (GlitchTip, Bugsink)
- Agent observability platforms that monitor agents (Langfuse, Laminar)
- AI SRE products that are SaaS-only (Rootly, incident.io)

Structural advantages already in place: RFC 9457, deterministic summaries,
built-in triage pipeline, SQLite simplicity.
